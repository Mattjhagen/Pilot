# Pilot Lending Architecture

## Philosophy

The Lending module establishes a trust-based origination and servicing flow. Unlike traditional banking apps that rely purely on FICO scores, Pilot evaluates the user's alternative `TrustScore` to generate `LoanOffer`s with personalized APRs.

## Components

### Data Models
Located in `LendingModels.swift`.
- `LoanOffer`: An array of pre-qualified terms generated by the `LendingService`.
- `LoanAgreement`: An active loan with an amortization schedule (`ScheduledPayment`).
- `LendingError`: Rejection states (e.g., `noOffersAvailable`).

### Service Layer
- `LendingService`: Protocol handling `prequalifyLoans`, `apply`, `fetchActiveLoans`, and `makePayment`.
- `MockLendingService`: Generates deterministic loan offers based on the Trust Score. 
  - Algorithm: `Base Rate (12%) - Discount (up to 8% based on Trust Score > 500)`.

### Managers
- `LendingManager`: An `@Observable` manager coordinating the fetching of Trust Scores, generating offers, originating agreements, and talking to the `MoneyService` to simulate disbursements.

### Interface
- `LendingOverviewView`: The main dashboard (replacing WealthView for this phase) showing active loans and pre-qualified offers.
- `OfferDetailView` & `ApplicationFormView`: The multi-step flow presenting loan metrics (principal, APR, fees) and simulating KYC/Payroll verification before approval.
- `LoanDetailView`: Tracks the amortization schedule and outstanding balance. Allows the user to simulate making a payment.

## Note on External Operations
In this phase, all KYC, underwriting, and money movement is mocked. Disbursing a loan triggers a console log in `LendingManager`, but could easily be wired to mutate a persistent SwiftData `Transaction` log in the future.
